资料来源：<https://www.zhihu.com/people/8b18484dd6461cbae1e8b49aa212ef0d>，实际跳转到 `苏慕白量化`（`a-ba-a-ba-61-28`）。

抓取时间：2026-05-29（Asia/Shanghai）。作者页显示 `文章36`，本笔记逐篇核对并整理 36 篇文章。正文页来源均为知乎专栏 `zhuanlan.zhihu.com`。

## 一句话结论

这位作者的核心不是“用复杂模型预测价格”，而是从市场结构、交易规则、费用、精度、速度、对手盘和容量里找可执行的正期望机会；AI 可以提高信息处理和研究效率，但真实交易执行必须落到工程状态机、延迟、风控和成本核算上。

对 `financePi` 来说，最值得学习的是：

- 先证明机会真实存在，再写复杂系统：每个提案必须能解释收益来源、对手方、成本、容量和失效条件。
- 把 Pi 放在“研究员/分析员/复盘员”位置，不让它直接裸连交易所；Python 层继续强制风控、下单、保护单和日志。
- 增强微观结构记录：maker 成交后的 markout、撤单延迟、订单簿新鲜度、资金费率、tick/step 规则、手续费和滑点。
- 不迷信 LLM 或日频机器学习预测；更适合做新闻/公告/社媒抽取、策略假设生成、复盘归因和异常提醒。
- 把套利机会拆成 Scanner、Monitor、Executor、Settler/Risk 四层，而不是让一个大模型直接“看行情下判断”。

## 作者画像

作者自述方向是做市、CTA、套利。文章大多围绕加密市场高频、做市、跨所套利、期现/资金费率套利、链上行情、低延迟工程，也穿插 A 股市场结构、AI/LLM 交易边界和个人经历。

需要注意：个别早期文章涉及安全攻防或灰黑产式增长案例。本文只吸收其“结构化找约束、流程化验证、规模化执行”的工程方法，不采纳任何违法、侵入、绕过风控或平台规则的做法。

## 方法论提炼

### 1. 先找商业结构，再谈代码和模型

作者反复强调，量化不是先堆高数、深度学习和工程洁癖，而是先找到真实可赚钱的结构：谁在亏、谁在被迫交易、什么规则造成错价、什么摩擦阻止别人吃掉这笔钱。代码和数学应该服务于机会验证，而不是替代机会本身。

落到 `financePi`：Pi 每次提出交易都应先回答：

- 收益来源是什么：趋势、均值回归、资金费率、价差、事件冲击，还是做市 spread？
- 对手方是谁：噪声交易者、慢资金、被动再平衡资金、情绪盘、交易所规则摩擦，还是自己想象出来的 alpha？
- 净收益是否扣除了手续费、滑点、资金费率、冲击成本和失败腿风险？
- 机会容量多大，持仓规模扩大后是否会消失？
- `invalid_if` 是什么，什么时候说明假设错了？

### 2. 套利不是“看到价差”，而是执行闭环

跨所、期现、预测市场、精度套利的共性是：理论价差只是第一步，真实收益取决于可成交深度、双腿同步、保证金、转账/结算、交易所故障、规则变化、单腿风险和资金占用。

落到 `financePi`：

- 目前 supervisor 仍应优先做单交易所、单账户可控执行；跨所/链上机会先做观察和告警，不直接扩 live 权限。
- 对套利候选机会加入 `net_edge_bps`、`max_size_by_depth`、`fee_bps`、`slippage_bps`、`funding_bps`、`execution_risk` 字段。
- 如果未来新增套利执行，必须同时新增双腿状态机、失败腿处理、日志、文档和测试。

### 3. 做市的关键是逆向选择，不是成交越多越好

做市文章的主线是：单纯提高成交率会吃到更多有毒订单；maker 要看成交后的 markout，而不是只看挂单成交量。好的做市系统要在高成交概率区域里识别“可吃成交”和“有毒成交”。

落到 `financePi`：

- 对每笔 maker 成交记录成交后 `1s/5s/30s/60s` 的 mid-price markout。
- 记录成交前盘口不平衡、近端/对端队列、短期成交方向、spread、波动、订单存活时间。
- 以后若启用 maker 入场，风控不能只判断价格偏离，还要判断盘口毒性和撤单时效。

### 4. 延迟和基础设施会改变策略边界

作者多篇文章讲服务器地区、AWS/阿里云、WebSocket、异步、HTTP session、多 IP、Rust/Go/Python 混合。核心不是盲目追求纳秒，而是知道策略需要什么级别的延迟，并把行情、订单、账户状态分层处理。

落到 `financePi`：

- 记录行情时间戳、接收时间、决策时间、下单时间、交易所确认时间。
- 对 WebSocket 行情设置 staleness 阈值；行情过旧时禁止开仓，只允许保护性动作。
- 订单回报不可丢，账户状态定时主动刷新；行情可以采用 latest-only 思路避免积压旧 tick。

### 5. AI/LLM 的位置是信息处理，不是裸执行

作者对 Alpha Arena、OpenClaw 和 AI 交易的判断很清晰：LLM 不擅长精确数值状态机和高频交易决策，但适合处理语言、新闻、公告、研报、代码、文档和流程推理。

这与本项目现有架构一致：Pi RPC 使用 `--no-tools` 和 `--no-context-files`，只读结构化 snapshot 并输出 JSON；真实执行由 Python 风控和受控 Binance 接口完成。

落到 `financePi`：

- 保持 Pi 不能直接调用 Binance、bash、文件编辑或裸 API。
- 扩大 Pi 的研究输入：新闻、公告、RSS、FRED、资金费率、OI、盘口、最近成交、账户状态。
- 缩小 Pi 的执行自由度：只允许结构化提案、解释、失效条件、复盘；执行参数由 Python 重新校验。

## 可落地到 financePi 的改进清单

### P0：保持现有安全边界

- Pi 继续只输出 JSON 决策和复盘，不直接接触 Binance API、shell、文件系统或 raw exchange API。
- 新增任何执行能力时，同步更新 Python 风控、日志、测试和文档。
- 外部文章中的黑客、绕限、灰产增长方法不进入项目实现。

### P1：增强交易提案的结构化字段

建议在 Pi 输出或 Python 归一化层增加这些字段：

```text
edge_source: trend | mean_reversion | funding | spread | event | market_making | other
counterparty_assumption: string
net_edge_bps: number
fee_bps: number
slippage_bps: number
funding_bps: number
capacity_notional: number
data_freshness_ms: number
execution_risk: low | medium | high
invalid_if: string
```

这些字段不应直接放行交易，而是给 Python 风控和日志提供可审计证据。

### P2：新增 microstructure / 执行质量日志

对每笔真实下单或成交记录：

- 下单前 BBO、spread、盘口不平衡、mark price、index price。
- 决策时间、下单时间、交易所 ack 时间、成交时间。
- maker 成交后的 `1s/5s/30s/60s` markout。
- 保护单创建/更新耗时，以及最后一个有效止损是否始终存在。

### P3：新增资金费率和精度机会监控

先只做只读监控，不直接执行：

- Binance 合约资金费率、预测资金费率、现货/合约价差。
- symbol 的 `tickSize`、`stepSize`、`minNotional`、contract size、价格精度。
- 记录理论价差、扣费后净价差、可成交深度和容量。

### P4：把 AI 用在信息优势，而不是价格神谕

Pi 可以更积极地做：

- 新闻、公告、监管、宏观数据的摘要和冲击路径分析。
- 对已有持仓的 thesis 复核、失效条件检查、复盘归因。
- 对策略 idea 生成“需要验证的数据清单”和“可能失败原因”。

Pi 不应该做：

- 仅凭 K 线和自然语言感觉给出高杠杆方向单。
- 用日频价格预测替代可执行回测。
- 因为 LLM 说“有机会”而跳过 Python 风控。
